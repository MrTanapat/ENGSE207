# Reflection

## Weekly Reflection: Monolithic Architecture

### 1. What did you learn?
ได้เรียนรู้วิธีการสร้างเว็บแบบ Monolithic (แบบมัดรวม) โดยใช้ Node.js กับ Express ครับ ทำให้เห็นภาพชัดเจนเลยว่าทั้ง 3 ส่วนหลักของการทำเว็บ คือหน้าบ้าน (HTML/JS), หลังบ้าน (Express Routes) และฐานข้อมูล (SQLite) มันเชื่อมต่อกันยังไงในโปรเจกต์เดียว และก็ได้ฝึกการเขียน API เพื่อให้หน้าเว็บดึงข้อมูลจาก Database มาโชว์ได้จริง

### 2. What are the main advantages of monolithic architecture? 
- เริ่มงานไว: ไม่ต้องตั้งค่าอะไรเยอะแยะ เปิดโปรเจกต์เดียวก็เริ่มเขียนโค้ดได้เลย จัดการทุกอย่างในที่เดียว
- รันง่าย: แค่สั่งรัน Server ตัวเดียว จบเลย ไม่ต้องคอยเปิดหลายๆ Service ให้วุ่นวาย
- จัดการข้อมูลง่าย: เพราะเราใช้ Database ก้อนเดียว ไม่ต้องกังวลเรื่องข้อมูลไม่ตรงกันระหว่างระบบ

### 3. What challenges did you encounter?
- โค้ดเริ่มรก: พอเริ่มเพิ่มฟีเจอร์เยอะขึ้น (เช่น เพิ่มปุ่มลบ, ปุ่มแก้ไข) ผมสังเกตว่าไฟล์ server.js มันเริ่มยาวเหยียด ถ้าเป็นโปรเจกต์จริงที่ใหญ่กว่านี้ คงจะแก้ยากและตาลายแน่นอนถ้าไม่แบ่งไฟล์ดีๆ (Spaghetti Code)
- พังทีเดียวดับหมด: ตอนที่ผมเขียนโค้ดหลังบ้านผิดนิดเดียว Server มันล่มไปเลย ทำให้หน้าเว็บใช้งานไม่ได้ไปด้วย ซึ่งนี่คือความเสี่ยงของระบบแบบนี้ครับ (Single Point of Failure)

### 4. When should you use monolithic architecture? 
- โปรเจกต์เริ่มแรก (Startup/MVP): ที่เน้นสร้างของให้เสร็จไวๆ เพื่อทดสอบไอเดีย
- ทีมเล็ก: ถ้ามี Dev แค่ไม่กี่คน การทำ Monolithic จะช่วยให้คุยกันง่าย ทำงานบนโค้ดชุดเดียวกันได้สะดวก
- แอปที่ไม่ซับซ้อน: ถ้าฟีเจอร์ไม่ได้เยอะหรือซับซ้อนมาก การแยก Service อาจจะทำให้งานยากเกินความจำเป็นครับ
